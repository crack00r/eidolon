## 3. v1.2 Feature 1: WhatsApp Channel

### Design Decision: Official Business API, NOT Baileys

The user explicitly specified the official WhatsApp Business API. This is the correct choice:

| Approach | Pros | Cons |
|---|---|---|
| **WhatsApp Business API** (chosen) | Official, stable, no ban risk, supports templates | Requires Meta business verification, per-conversation pricing |
| Baileys (reverse-engineered) | Free, no verification | Ban risk (ToS violation), fragile, requires phone pairing |

### Architecture

```
WhatsApp Cloud API (Meta)
    |
    | Webhook (HTTPS POST to gateway)
    v
Gateway webhook handler (/webhooks/whatsapp)

    v

    |

    v
CognitiveLoop -> Claude Code -> response
    |
    | routeOutbound
    v
WhatsAppChannel.send()
    |
    | POST to WhatsApp Cloud API
    v
User receives reply on WhatsApp
```

### WhatsApp-Specific Behavior

1. **Webhook verification:** WhatsApp sends GET request with `hub.verify_token` and `hub.challenge`. Handler must echo back the challenge.
2. **Signature verification:** Every POST has `X-Hub-Signature-256` header. Verify with HMAC-SHA256 using app secret.
3. **24-hour window:** WhatsApp restricts messages to within 24 hours of last user message. After 24 hours, only template messages are allowed. The channel must track last-message timestamps.
4. **Message formatting:** WhatsApp uses a subset of markdown: `*bold*`, `_italic_`, `~strikethrough~`, `` `code` ``, ``` ```code block``` ```. The formatter converts Claude's full markdown to this subset.
5. **Phone number format:** All phone numbers must be in E.164 format (e.g., `+491234567890`). The allowlist uses this format.
6. **Media handling:** Media must be uploaded to WhatsApp's Media API first, then sent by media ID. Downloads go through the same API.
7. **Rate limiting:** WhatsApp Cloud API has a 1,000 messages/second limit per phone number. Per-user rate limiting at 30/minute mirrors the Telegram pattern.

## 4. v1.2 Feature 2: Email Channel

### Architecture

Email is fundamentally different from real-time messaging channels. It is asynchronous, supports long-form content, threading, attachments, and has no streaming.

```
IMAP Server (incoming mail)
    |
    | Poll every N seconds (configurable)
    v
EmailChannel.poll()
    |
    | Parse MIME, extract text/attachments
    v
routeInbound -> EventBus
    |
    v
CognitiveLoop -> Claude Code -> response
    |
    | routeOutbound
    v
EmailChannel.send()
    |
    | SMTP with proper threading headers
    v
User receives reply email
```

### Email-Specific Behavior

1. **IMAP polling:** The channel polls for new messages at `pollIntervalMs` intervals using an IDLE-capable connection when supported, falling back to regular polling.
2. **Threading:** Replies use `In-Reply-To` and `References` headers to maintain email threads. The `Message-ID` of each sent email is stored for future reference.
3. **HTML rendering:** Claude's markdown output is converted to styled HTML email using a clean template. A plain-text fallback is always included.
4. **Attachment handling:** Inbound attachments are extracted and passed to Claude as `MessageAttachment` objects. Outbound attachments are MIME-encoded.
5. **Subject lines:** New conversations get `[Eidolon] <topic>`. Replies preserve the original subject with `Re:` prefix.
6. **Sender allowlist:** Only emails from `allowedSenders` are processed. Others are silently ignored (no bounce).
7. **Not a streaming channel:** The `capabilities.streaming` flag is `false`. Full responses are sent as complete emails.

## 5. v1.2 Feature 3: OpenTelemetry Integration

### Architecture

OpenTelemetry (OTel) adds distributed tracing and structured metrics export. This complements the existing Prometheus metrics (`packages/core/src/metrics/prometheus.ts`) with request-level tracing across service boundaries.

```
Core Daemon                    GPU Worker              OTel Collector
    |                              |                        |
    |--- trace: user.message -->   |                        |
    |    span: cognitive-loop      |                        |
    |    span: memory-search       |                        |
    |    span: claude-session      |                        |
    |        span: tts-request --->|                        |
    |        <--- tts-response ----|                        |
    |    span: memory-extract      |                        |
    |    span: channel-send        |                        |
    |                              |                        |
    |------- export traces --------|-------->                |
    |------- export metrics -------|-------->                |
```

### Design Decisions

1. **OTel SDK for TypeScript (`@opentelemetry/sdk-node`):** The standard approach. Provides auto-instrumentation for HTTP, and manual span creation for business logic.
2. **Complement, do not replace Prometheus:** The existing `MetricsRegistry` in `prometheus.ts` continues to serve the `/metrics` endpoint. OTel adds trace context propagation and distributed tracing that Prometheus cannot provide.
3. **Optional dependency:** OTel is opt-in via config. When disabled, all tracing calls are no-ops.
4. **GPU Worker tracing:** The Python GPU Worker uses `opentelemetry-python` SDK. Trace context is propagated via HTTP headers (`traceparent`).

export interface ITracer {
  /** Create a span and run the function within it. */
  withSpan<T>(name: string, attributes: Record<string, string | number | boolean>,
              fn: () => Promise<T>): Promise<T>;

  /** Create a span manually (caller must end it). */
  startSpan(name: string, attributes?: Record<string, string | number | boolean>): ISpan;

  /** Get headers for propagating trace context to external services. */
  getTraceHeaders(): Record<string, string>;

  /** Extract trace context from incoming headers. */
  extractContext(headers: Record<string, string>): void;
}

export interface ISpan {
  setAttribute(key: string, value: string | number | boolean): void;
  setStatus(code: "ok" | "error", message?: string): void;
  addEvent(name: string, attributes?: Record<string, string | number | boolean>): void;
  end(): void;
}

/** No-op tracer for when OTel is disabled. */
export class NoopTracer implements ITracer {
  async withSpan<T>(_name: string, _attrs: Record<string, string | number | boolean>,
                    fn: () => Promise<T>): Promise<T> { return fn(); }
  startSpan(): ISpan { return new NoopSpan(); }
  getTraceHeaders(): Record<string, string> { return {}; }
  extractContext(): void {}
}

## 7. v2.0 Feature 1: Plugin System

### Architecture

The plugin system enables third-party extensions that can:
- Add new CLI commands
- Add new gateway RPC methods
- Register EventBus listeners
- Contribute memory extractors
- Add new channel implementations
- Extend configuration with custom schemas

Plugins are loaded at daemon startup from a configured directory.

```
~/.eidolon/plugins/
    |
    +-- eidolon-plugin-todoist/     # npm package
    |     package.json              # { "eidolon": { "type": "plugin" } }
    |     dist/index.js             # compiled plugin entry
    |
    +-- eidolon-plugin-notion/
          package.json
          dist/index.js
```

### Design Principles

1. **Plugins are npm packages.** Standard packaging, versioning, dependency management.
2. **Plugins cannot modify core code.** They extend through defined extension points.
3. **Plugins run in the same process.** No IPC overhead. Sandboxing is opt-in via config.
4. **Plugins have a lifecycle.** `initialize()`, `start()`, `stop()`, `destroy()`.
5. **Plugins declare capabilities.** What extension points they use, what permissions they need.

### Plugin Loading Sequence

```
1. Scan plugins directory for directories with package.json
2. Read package.json, extract eidolon manifest
3. Validate manifest: name, version, eidolonVersion compatibility
4. Check permissions against security policy
5. Load plugin module (dynamic import)
6. Call plugin.initialize(context) with sandboxed PluginContext
7. Call plugin.start()
8. Register extension points (channels, RPC handlers, event listeners)
9. On daemon shutdown: call plugin.stop() then plugin.destroy()
```

### Plugin Sandboxing

Plugins declare required permissions. The sandbox enforces these:

```typescript
// packages/core/src/plugins/sandbox.ts
export function createSandboxedContext(
  fullContext: PluginContext,
  permissions: readonly PluginPermission[],
): PluginContext {
  return {
    logger: fullContext.logger, // always available
    config: permissions.includes("config:read") ? fullContext.config : {} as EidolonConfig,
    eventBus: {
      on: permissions.includes("events:listen")
        ? fullContext.eventBus.on
        : () => { throw new Error("Permission denied: events:listen"); },
      emit: permissions.includes("events:emit")
        ? fullContext.eventBus.emit
        : () => { throw new Error("Permission denied: events:emit"); },
    },
    memory: {
      search: permissions.includes("memory:read")
        ? fullContext.memory.search
        : () => Promise.resolve(Err(createError(ErrorCode.GATEWAY_AUTH_FAILED, "Permission denied"))),
      store: permissions.includes("memory:write")
        ? fullContext.memory.store
        : () => Promise.resolve(Err(createError(ErrorCode.GATEWAY_AUTH_FAILED, "Permission denied"))),
    },
    gateway: {
      registerHandler: permissions.includes("gateway:register")
        ? fullContext.gateway.registerHandler
        : () => { throw new Error("Permission denied: gateway:register"); },
    },
    channels: {
      register: permissions.includes("channel:register")
        ? fullContext.channels.register
        : () => { throw new Error("Permission denied: channel:register"); },
    },
    cli: {
      registerCommand: permissions.includes("cli:register")
        ? fullContext.cli.registerCommand
        : () => { throw new Error("Permission denied: cli:register"); },
    },
  };
}
```

## 8. v2.0 Feature 2: Local LLM Support

### Architecture

The current `IClaudeProcess` interface is tightly coupled to Claude Code CLI semantics (workspace dirs, session IDs, tool whitelists). For Local LLM support, we need a broader abstraction.

**Strategy: Introduce `ILLMProvider` as the parent abstraction. `IClaudeProcess` becomes one implementation.**

```
                    ILLMProvider
                   /      |      \
                  /       |       \
    ClaudeProvider  OllamaProvider  LlamaCppProvider
    (wraps           (HTTP API)     (subprocess)
     IClaudeProcess)
```

### Design Decisions

1. **`ILLMProvider` is NOT a drop-in replacement for `IClaudeProcess`.** Claude Code provides tools (Bash, Edit, Glob, etc.) that local LLMs do not. The provider abstraction handles text generation only. Tool use is handled by a separate `ToolExecutor` layer.
2. **ClaudeProvider remains the primary provider.** Local LLMs are for cost reduction on specific tasks (memory extraction, relevance filtering, dreaming).
3. **Model routing by task type.** The Cognitive Loop's Evaluate phase selects the provider based on task requirements.
4. **Ollama is the primary local backend.** It has the broadest model support, runs on all platforms, and has a simple HTTP API.

| Local LLM -> Memory Extractor | Extractor accepts ILLMProvider | Interface change |
| Local LLM -> Cognitive Loop | Loop uses model router | Interface change |

## 12. Risk Register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| WhatsApp Business API verification delays | Medium | High (blocks S1) | Start verification process immediately. S2 (Email) can proceed independently. |
| WhatsApp per-conversation pricing concerns | Low | Low | Free tier: 1,000 conversations/month. Personal use stays well under this. |
| Bun compatibility with email libraries | Medium | Medium | Test `imapflow`, `nodemailer`, `mailparser` on Bun before sprint. Fallback: use Bun's built-in `fetch` for SMTP/IMAP raw protocol. |
| OTel SDK Bun compatibility | Low | Medium | HTTP exporter works. gRPC may not. Default to HTTP protocol. |
| Plugin sandboxing is insufficient | Medium | High | Start with permission-based sandbox. Consider Worker threads or separate process for untrusted plugins in v2.1. |
| Local LLM tool calling quality | High | Medium | Not all local models support function calling well. The ToolExecutor should handle malformed tool calls gracefully. Restrict tool access for local models. |
| Local LLM context window limits | Medium | Medium | Many local models have 4K-8K context. Memory injection must be truncated. Router should prefer Claude for tasks requiring large context. |
| ILLMProvider abstraction leaks | Medium | Medium | Claude has unique features (project directories, session resumption) that local LLMs lack. Keep `IClaudeProcess` for Claude-specific features. `ILLMProvider` is for generic text generation only. |
